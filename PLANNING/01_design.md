## 1. The build-target taxonomy

**8 ordered categories** (the spine reads as a learning order, not alphabetic вҖ” readers should sense progression). Within each category, build targets appear in difficulty order.

### 1. Foundations вҖ” *the machinery before the model*
1.1 Tokenizer (BPE)
1.2 Embedding layer
1.3 Autograd / micro-framework
1.4 Backprop by hand on an MLP

### 2. The Model вҖ” *attention to a full small LM*
2.1 Attention from scratch (scaled dot-product + multi-head)
2.2 Positional encodings (sinusoidal + RoPE)
2.3 Transformer block (LN + MHA + MLP + residuals)
2.4 A small GPT (full pipeline at toy scale)
2.5 A small encoder LM / BERT-style
2.6 Mixture-of-Experts block
2.7 State-space / Mamba-style block

### 3. Training вҖ” *how models learn*
3.1 Training loop (data вҶ’ loss вҶ’ optimizer step вҶ’ eval)
3.2 Optimizer (SGD вҶ’ Adam вҶ’ AdamW)
3.3 Learning-rate schedule (warmup + cosine)
3.4 Mixed precision + gradient accumulation
3.5 Supervised fine-tuning (SFT) of a small base
3.6 Reward model
3.7 DPO / preference tuning
3.8 PPO / GRPO-style RL loop

### 4. Inference вҖ” *making a trained model useful*
4.1 Sampling (greedy / temperature / top-k / top-p)
4.2 KV cache
4.3 Minimal inference engine (generate loop, batching)
4.4 Speculative decoding
4.5 Quantization (INT8 / INT4)

### 5. Retrieval вҖ” *external memory for LMs*
5.1 BM25 from scratch
5.2 Vector search (brute force вҶ’ HNSW)
5.3 Hybrid search (BM25 + vector)
5.4 Reranker (cross-encoder)
5.5 End-to-end RAG pipeline (no frameworks)

### 6. Agents вҖ” *LMs that take actions*
6.1 Agent loop (reason вҶ’ act вҶ’ observe)
6.2 Tool / function-calling layer
6.3 Agent memory (short + long term)
6.4 Multi-agent orchestration
6.5 Coding agent (file edits + shell)

### 7. Evaluation вҖ” *knowing if it actually works*
7.1 Eval harness (task вҶ’ prompt вҶ’ score)
7.2 LLM-as-judge
7.3 Calibration / hallucination check

### 8. Beyond Text вҖ” *vision, audio, multimodal*
8.1 Diffusion model (DDPM)
8.2 CLIP-style contrastive model
8.3 Speech recognition (CTC)

**Totals:** 8 categories В· 40 build targets В· ordered by difficulty within each.

**Within-category ordering rule:** prerequisites before applications. A reader who works the targets top-to-bottom in any category should never hit a target whose entries assume something later in the same category.

## 2. Entry schema

Every build target is one entry. Source-of-truth lives in `entries/<category>.yaml`; the README is regenerated by `build_readme.py`.

### YAML schema

```yaml
- id: tokenizer-bpe                      # kebab-case, unique repo-wide
  category: foundations                  # one of the 8 category slugs
  order: 1                               # display order within the category
  title: "Tokenizer (BPE)"
  what_you_build: "A byte-pair-encoding tokenizer that turns raw text into a vocabulary of subword tokens вҖ” the same idea that powers GPT-class tokenizers."
  teaches: ["subword tokenization", "vocab construction", "merges"]
  understanding: "Why models see tokens, not characters or words вҖ” and why that single decision shapes everything downstream."
  difficulty: 2                          # 1, 2, or 3 (rendered as вҳ… / вҳ…вҳ… / вҳ…вҳ…вҳ…)
  guides:
    - title: "minbpe"
      url: "https://github.com/karpathy/minbpe"
      author: "Andrej Karpathy"
      kind: code                         # code | written | video | book | course | original
      language: "Python"                 # primary language; "Mixed" if multi
      cost: free                         # free | paid
      from_scratch_evidence: "No tokenizer libraries imported; merges implemented by hand; companion video walks through every step."
      verified_on: 2026-05-14            # ISO date the entry was last verified live
      verifier: "bettyguo"
      notes: "Pair with the lecture video for full effect."
  see_also:
    - "https://huggingface.co/learn/nlp-course/chapter6/5"  # *not* counted as a primary entry; just adjacent reading
  gap: false                              # true if no good guide found вҖ” target ships as a gap
  original_planned: false                 # true if `originals/` should fill this
```

### Required fields (validator enforces)

`id` В· `category` В· `order` В· `title` В· `what_you_build` В· `teaches` В· `understanding` В· `difficulty` В· `guides` (or `gap: true`).

### Per-guide required fields

`title` В· `url` В· `author` В· `kind` В· `language` В· `cost` В· `from_scratch_evidence` В· `verified_on` В· `verifier`.

The `from_scratch_evidence` field is the single most important field in the repo вҖ” it is the human-readable proof that this guide actually teaches by reimplementation. The validator requires it to be non-empty and >= 20 characters. The hostile reviewer in Phase 6 reads only these fields.

### Rendered README format (per entry, generated)

```
#### Tokenizer (BPE) вҳ…вҳ…

**What you build:** A byte-pair-encoding tokenizer that turns raw text into a vocabulary of subword tokens вҖ” the same idea that powers GPT-class tokenizers.

**What you'll understand after:** Why models see tokens, not characters or words вҖ” and why that single decision shapes everything downstream.

- [**Python**: _minbpe_](https://github.com/karpathy/minbpe) вҖ” Andrej Karpathy В· code В· free
```

Compact but slightly richer than `build-your-own-x` (which is pure one-liners). The richer single-line "understanding" promise is what differentiates this index from a list of links.

### Tags rendered

- Difficulty: `вҳ… / вҳ…вҳ… / вҳ…вҳ…вҳ…`
- `code` В· `written` В· `video` В· `book` В· `course` В· `original` (kind)
- `free` В· `paid`
- `[original]` marker is bolded for `originals/` entries

## 5. Tooling design

### `tools/validate_entries.py`
- Loads every `entries/*.yaml`.
- Schema check: all required fields present (per В§2).
- Cross-file uniqueness check on `id`.
- Category slugs must match filename order prefix (e.g. `category: foundations` only in `1-foundations.yaml`).
- Each guide must have `from_scratch_evidence` вүҘ 20 chars; `verified_on` must be a real ISO date вүӨ today; `kind`, `cost`, `language` from controlled vocabularies.
- Duplicate-URL detection across all entries.
- Either `gap: true` *or* at least one guide is required (not both).
- If `original_planned: true`, expects a corresponding file in `originals/`.
- Exits 1 on any failure; prints a structured diff of what's wrong and where.

### `tools/linkcheck.py`
- Walks every URL in `entries/*.yaml`, `originals/*.md`, and `README.md`.
- Concurrent HEAD (fallback GET) with a polite delay and user-agent.
- Treats 403/429 from rate-limiting as warnings (not failures) when retry hints succeed on backoff.
- Output: a Markdown report `tools/linkcheck-report.md` listing each URL's status.
- Two modes: `--quick` (10s timeout, only changed files in CI) and `--full` (60s timeout, all URLs; used in scheduled workflow).
- Exits 1 if any verified-on entry's URL is hard-broken (4xx/5xx persistent).

### `tools/build_readme.py`
- Loads `README.md.tmpl`, fills in:
  - the TOC,
  - the generated index (every entry rendered per В§2),
  - the `Originals` block (listed from `originals/README.md` frontmatter),
  - the "Last updated" badge (max `verified_on` across entries),
  - the changelog "Recent additions" (top 5 most recent `verified_on`).
- Writes `README.md`.
- CI verifies the committed `README.md` matches the regenerated one (drift check).

### `tools/requirements.txt`
```
pyyaml>=6.0
jinja2>=3.1
httpx>=0.27
click>=8.1
```

### CI workflows

`.github/workflows/ci.yml` (on every PR + push to `main`):
1. Set up Python 3.12.
2. `pip install -r tools/requirements.txt`.
3. Run `validate_entries.py` (must pass).
4. Run `build_readme.py`; `git diff --exit-code README.md` (no drift).
5. Run `linkcheck.py --quick` on URLs touched by this PR.

`.github/workflows/linkcheck.yml` (scheduled weekly + on-demand):
1. Run `linkcheck.py --full`.
2. If failures, open / update an issue titled "Link-check failures вҖ” YYYY-WW".

## 6. Contribution flow

`CONTRIBUTING.md` requires every PR adding an entry to:
1. Open `entries/<category>.yaml`, add a YAML block.
2. Run `python tools/validate_entries.py` locally; CI will re-run.
3. Run `python tools/build_readme.py`; commit the regenerated README.
4. Fill in `from_scratch_evidence` with one concrete sentence pointing to *where in the guide* the from-scratch property is visible (e.g., "Implements attention from `numpy` primitives in section 3"; "No `transformers` import anywhere in the repo").

`.github/PULL_REQUEST_TEMPLATE.md` includes:
- [ ] I have run `validate_entries.py` and `build_readme.py` locally.
- [ ] The `from_scratch_evidence` for each new guide cites a concrete location.
- [ ] I confirm each guide is currently working (the code runs / the tutorial loads).
- [ ] I have NOT linked a guide whose primary teaching method is "use library X."

`.github/ISSUE_TEMPLATE/not-from-scratch.md` exists specifically so the community can report mislabeled entries; this is the curation safety valve.

## 7. Verification plan

For Phase 3 (population) and reaffirmed in Phase 5 (launch-readiness):
- Every guide URL is fetched and confirmed live.
- The author / title are confirmed against the source.
- The from-scratch evidence is recorded.
- The currency check: README/last-commit dated within the last ~24 months, *or* the technique itself is timeless (BM25, backprop, autograd). Anything older than 24 months that pretends to teach a fast-moving area (RAG, agents, inference engines) is rejected unless the technique itself hasn't drifted.
- The verifier (`bettyguo`) and `verified_on` date are written to each entry.
- Outcome вҖ” pass / reject + reason вҖ” is logged in `PLANNING/03_verification_log.md`.
